# Design Brief Writer

A design brief gives a designer the context, constraints, and success criteria they need to produce the right design - without prescribing the solution.

## What Goes in a Design Brief

Apply this skill to: $ARGUMENTS


A good design brief answers:
1. **Why are we building this?** (User problem and business goal)
2. **Who are we designing for?** (Primary persona with relevant behavioral context)
3. **What must the design accomplish?** (Functional requirements expressed as user outcomes)
4. **What constraints exist?** (Technical, time, brand, platform, accessibility)
5. **What does success look like?** (How will we know the design works?)
6. **What is out of scope?** (What should not be included)
7. **What context should the designer have?** (Research insights, competitive examples, existing patterns to follow or break)

## What Does NOT Go in a Design Brief

A brief should NOT:
- Prescribe specific UI components or layouts (that is the designer's job)
- Include wireframes created by the PM (leads to "fill in the pixels" work, not design thinking)
- List features without explaining the user problem they solve
- Be so vague that a designer cannot determine what success looks like

## Output Template

---
**DESIGN BRIEF: [Feature Name]**
**Author:** [PM Name]
**Date:** [Date]
**Designer:** [Name if assigned]
**Target Delivery:** [Date for first design review]

**Problem Statement**
[The user problem this design will solve, with evidence from research or data]

**User Context**
Primary persona: [Name from persona work]
Relevant behavioral context: [How this persona will encounter and use this feature - device, frequency, context of use]

**Design Goals**
1. [User outcome - e.g., "A first-time user can complete X without assistance"]
2. [User outcome]
3. [Business outcome - e.g., "Reduce support tickets about X by 40%"]

**Functional Requirements** (what the design must enable, expressed as user actions)
- [User can do X]
- [User can do Y]
- [System provides Z feedback when...]

**Constraints**
Technical: [Any technical limitations that affect design options]
Brand: [Brand guidelines or existing design system requirements]
Platform: [iOS, Android, Web, or all - with any platform-specific constraints]
Accessibility: [WCAG level required]
Timeline: [Hard dates]

**Out of Scope**
- [Explicitly excluded scope to prevent scope creep]

**Research and Context**
[Key insights from user research, relevant customer quotes, competitive examples worth reviewing]

**Existing Patterns to Follow**
[Any existing UI patterns in the product this feature should be consistent with]

**Intentional Departures**
[Any cases where this feature should deliberately diverge from existing patterns and why]

**Success Criteria**
[How the PM will evaluate whether the design works - usability test benchmark, A/B test metric, etc.]
