# Design Thinking for Product Managers

Design thinking is a human-centered, iterative approach to innovation. It is not just for designers - it is a PM's sharpest tool for attacking ambiguous problems.

## The Double Diamond Framework

Apply this skill to: $ARGUMENTS


**Diamond 1: Discovery (Diverge then Converge)**
- Diverge: Explore the full problem space (research, observation, immersion)
- Converge: Define the right problem to solve (insight synthesis, HMW framing)

**Diamond 2: Development (Diverge then Converge)**
- Diverge: Generate as many solution ideas as possible (ideation, brainstorming)
- Converge: Select, prototype, and test the best solutions (prototyping, testing, iteration)

The key insight: Most teams rush to Diamond 2 before doing Diamond 1. They solve the wrong problem well.

## How Might We (HMW) Statements

HMW statements reframe a problem into an opportunity for ideation. They are specific enough to be actionable but open enough to allow diverse solutions.

Formula: "How might we [verb] [outcome] for [user] so that [higher-level goal]?"

Too broad: "How might we improve our product?"
Too narrow: "How might we add a filter to the search results page?"
Just right: "How might we help returning users find their most relevant content faster?"

Start with user pain points from research, then write 5-10 HMW statements before selecting 1-3 to ideate on.

## Ideation Techniques

**Crazy 8s**: Each participant sketches 8 ideas in 8 minutes (forces quantity over quality)
**SCAMPER**: Substitute, Combine, Adapt, Modify, Put to other uses, Eliminate, Reverse
**Analogous inspiration**: How does a completely different industry solve a similar problem?
**Constraint ideation**: What if you had to solve this with zero budget? In one day? With no new features?

## Prototype-Test-Learn Loop

1. **Prototype**: Build the minimum artifact needed to test your key assumption (sketch, mockup, role-play, paper prototype)
2. **Test**: Put it in front of 5 real users (5 users reveals ~85% of usability issues per Nielsen Norman Group research)
3. **Learn**: What did users do vs. what you expected? What surprised you?
4. **Iterate**: Update the prototype based on what you learned
5. Repeat until confident in the solution

## Design Sprint (5-Day Format, Jake Knapp)

- **Monday**: Define the long-term goal, map the problem, pick a target
- **Tuesday**: Review existing solutions, sketch competing approaches
- **Wednesday**: Critique sketches, decide on one direction, storyboard
- **Thursday**: Build a realistic prototype (only what is needed for Friday)
- **Friday**: Test with 5 target users, synthesize insights
